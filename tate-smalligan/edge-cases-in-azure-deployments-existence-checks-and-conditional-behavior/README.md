# Edge Cases in Azure Deployments: Existence Checks and Conditional Behavior

## Abstract

Infrastructure‑as‑code systems are designed to be declarative and idempotent, yet real‑world cloud deployments frequently expose edge cases that challenge these assumptions. A common example is the need for existence checks: determining whether a resource already exists before deciding how—or whether—to proceed with a deployment. While this appears to be a simple requirement, the original request for existence checks in Bicep arose from scenarios where idempotency alone was insufficient, such as resource providers returning errors when a resource already existed rather than treating the operation as a no‑op.
This talk examines why existence checks are necessary, when they are unavoidable, and what makes their behavior difficult to reason about safely in practice. Using Azure Resource Manager and Bicep as primary examples, we will explore challenges including inconsistent resource provider behavior and APIs that do not strictly adhere to expected contracts. We will also discuss the additional mechanisms introduced to handle these edge cases—such as wait‑and‑retry patterns and deployment scripts to enable controlled conditional execution. The session concludes with a practical demonstration of how to handle common edge cases encountered in real‑world Azure deployments, with existence checks as a central example.
